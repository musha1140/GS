How v2 works. The chosen MIDI file is a constant, so its parsed note sequence is a constant too. Each note contributes three bytes — midi number, velocity, ticks — forming a deterministic keystream. Your input (a typed message, or any file's raw bytes) is XORed against that keystream, cycling circularly when the lengths differ, which they always do. XOR twice with the same song and you get the original back, bit for bit: a lossless chop. Two things reconstruct the message — the .slayy file and the song. An AES-256-GCM seal (Web Crypto, PBKDF2) wraps the payload for one-click decompiling with integrity, and the decompiler re-derives the keystream from the song to prove the bits check out. A finished dump can also collapse into a syzygy QR — tap it for the scannable form (QR's own Reed–Solomon error correction carries the payload) and whoever scans it watches the code break down through the binary rain until the 64 cells spell the message out. The passphrase lives in this page's source: this is art, not security.

This project is an extension of the concept from Slayybit Image Embed, which explores lazy (look for .slayy at the end of a string) ways of embedding messages within images using MIDI files. The message is dynamically altered by the MIDI file being played, using the XOR operation for real-time binary transformation.

The core idea uses a MIDI synth to alter the message as the song plays, inspired by the MIDI Synth design and built on the Tone.js MIDI library. As the song plays, the text is XORed with the notes in real time, creating a unique binary representation that is visualized and embedded within an SVG image — saved together as a .slayy file encapsulating both visual and encrypted data.

The term "Slayy" phonetically represents 'Ghidra's Sleigh', openly available on the NSA's GitHub, which uses various techniques to recover lost binary data — embedding and transforming data through encryption and visualization. This project is a nod to the importance of secure and hidden data transmission.
